==================================================
IP BOX / INTELLECTUAL PROPERTY DOCUMENTATION RULE
==================================================

This project may be used as supporting technical documentation for an IP Box application.

For that reason, the development assistant must help maintain a clear, factual and auditable development record.

Important ownership and authorship context:

- The product idea, functional concept, business use case, technical direction and acceptance decisions come from the project owner: Andrés Velasco / ANVEL Consulting / ERP Utilities Services Ltd.
- Development tools, including AI-assisted tools where applicable, are used as productivity support, not as the creator or owner of the software.
- The documentation must reflect that the user defines the requirements, validates the functionality, decides the architecture direction, tests the result, and accepts or rejects changes.
- Do not describe the project as “generated by ChatGPT” or “created by AI”.
- Correct wording should be similar to:
  “The project was designed, directed, reviewed and validated by the project owner, using AI-assisted development tools to accelerate implementation.”
- Do not fabricate evidence, dates, time spent, test results or decisions.
- All documentation must remain truthful, technically accurate and auditable.

The development assistant must create and maintain an IP Box documentation folder if it does not already exist:

/docs/ip-box/

This folder should contain or be prepared to contain:

1. development_log.md
   A chronological record of meaningful development steps.
   Each entry should include:

- date
- feature/module worked on
- technical objective
- implementation summary
- files changed
- user/business reason
- validation status
- open limitations
- whether development tooling support was used
- human decision/review notes

2. architecture_decisions.md
   A record of important technical decisions.
   Each decision should include:

- decision title
- context
- options considered
- selected option
- reason for selection
- expected benefit
- risks/limitations
- relation to SAP/ABAP/Eclipse/ADT use case
- whether this decision was made by the project owner

3. feature_register.md
   A structured list of implemented and planned features.
   Each feature should include:

- feature name
- phase
- status
- business purpose
- technical description
- files/modules involved
- validation method
- IP relevance / productivity relevance

4. evidence_register.md
   A register of evidence that can later support IP Box documentation.
   Examples:

- screenshots to be taken manually
- demo scenarios
- test cases
- commit references
- build logs
- changelog entries
- design notes
- manual validation notes

5. human_contribution_log.md
   A clear record showing the role of the project owner.
   It should document:

- requirements provided by the owner
- functional decisions made by the owner
- SAP/ABAP domain knowledge contributed by the owner
- review/acceptance decisions
- rejected or modified implementation suggestions
- manual testing performed by the owner
- business rationale provided by the owner

6. ai_assistance_disclosure.md
   A transparent explanation of development tooling support.
   It should state:

- development tools may be used to accelerate coding, drafting, refactoring and documentation.
- implementation outputs are reviewed and accepted by the project owner.
- development tools do not own the software.
- development tools do not autonomously define the business requirements.
- development tools do not replace human validation.
- The final responsibility for design, acceptance and use remains with the project owner.

7. test_and_validation_log.md
   A record of manual and automated validation.
   Each entry should include:

- test date
- tested feature
- test scenario
- expected result
- actual result
- status
- issues found
- follow-up required
- reviewer/validator

Documentation maintenance rule:

Whenever the development assistant implements or changes a feature, it must also update the relevant IP Box documentation files.

At minimum, every meaningful implementation must update:

- README.md
- CHANGELOG.md
- /docs/ip-box/development_log.md
- /docs/ip-box/feature_register.md

When a technical decision is made or clarified, also update:

- /docs/ip-box/architecture_decisions.md

When a feature is tested or manual validation is described, also update:

- /docs/ip-box/test_and_validation_log.md

When the user provides business/functional reasoning, SAP/ABAP-specific requirements, or accepts/rejects a proposed design, also update:

- /docs/ip-box/human_contribution_log.md

When development tooling disclosure is relevant to the workflow, keep:

- /docs/ip-box/ai_assistance_disclosure.md

up to date.

Important:
The purpose is not to create artificial legal claims.
The purpose is to preserve a truthful technical and development record showing:

- what was built
- why it was built
- which technical problems it solves
- which decisions were made by the project owner
- how development tooling support was used
- how the software evolved over time
- how the tool may improve productivity in SAP ABAP/Eclipse consulting work

Do not invent productivity percentages, financial values, tax conclusions or legal statements.
If such numbers are needed later, leave placeholders for the project owner or tax advisor.

Add a disclaimer in the IP Box documentation:
“This documentation is a technical development record. It is not legal or tax advice. Final IP Box eligibility, valuation and income attribution must be reviewed by qualified advisors.”

IP Box documentation should support the position that:

- the tool is proprietary software developed under the direction of the project owner
- the tool solves a specific technical/business problem in SAP ABAP/Eclipse consulting workflows
- the tool has a documented architecture and development history
- the tool includes original requirements, design decisions, testing and validation
- development tooling was used as implementation support, while ownership, direction, review and acceptance remained with the project owner

GitHub publishing rule:

- After completing each key task or milestone, the development assistant must commit and push the relevant changes to the correct GitHub repository for this project: https://github.com/Andresvelascofdez/AbapEclipsePlugin.
- Before pushing, the development assistant must verify that the local git remote points to Andresvelascofdez/AbapEclipsePlugin.
- The development assistant must never push this project's changes to any unrelated repository.
- If the git remote is missing or points to the wrong repository, the development assistant must stop, report the issue, and ask the project owner to confirm the safe repository setup before pushing.
